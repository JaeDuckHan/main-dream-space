# 럭키다낭 성장 전략 + 플래너 기능 강화 설계

## 개요

사이트 오픈 직후 1주일 런칭 액션 플랜과 플래너 기능 강화를 통해 신규 사용자 유입과 수익 모델을 동시에 검증한다.

**사이트 최종 목적:** 베트남 장기체류를 고민하는 한국인이 가장 먼저 찾는 의사결정 허브. 사용자는 도구와 콘텐츠를 무료로 사용하고, 업체는 유료 노출을 통해 수익이 발생하는 양면 플랫폼(Two-sided Platform).

---

## Part 1. 1주일 런칭 액션 플랜

### 목적

방향을 확정하지 않은 상태에서 1주일을 실험 주간으로 운영한다. 7일 후 실제 데이터(유입 경로, 플래너 사용 여부, 애필리에이트 클릭, 커뮤니티 가입)를 보고 2주차 방향을 결정한다.

### 핵심 전략: 플래너 = 콘텐츠 생성기

플래너를 직접 사용하는 과정 자체가 블로그/스레드 소재가 된다. 경쟁 블로그는 개인 경험담이지만, 럭키다낭은 **실제 데이터 + 실제 도구**가 차별점이다.

### 일별 액션 플랜

#### Day 1 (월) — 플래너 직접 사용 + 커뮤니티 시딩

**플래너 검증:**
- "30대 커플, 다낭, 30일, 예산 300만원" 시나리오로 플래너 처음부터 끝까지 직접 작성
- 막히는 UX, 버그, 빠진 체크리스트 항목 기록
- 애필리에이트 링크(Agoda, Booking)가 플래너 체크리스트에서 자연스럽게 노출되는지 확인

**커뮤니티 시딩 (관리자 계정으로):**
- 질문 글 1개: "다낭 한달살기 시작하려는데 숙소는 어디가 좋을까요?"
- 정보 글 1개: "다낭 한달살기 필수 준비물 23가지 정리"
- 리뷰 글 1개: "미케비치 근처 한달살기 후기 (실제 생활비 포함)"

방문자가 처음 왔을 때 빈 게시판이면 즉시 이탈한다. 최소 3개 글로 활성화된 것처럼 보이게 한다.

#### Day 2 (화) — 콘텐츠 소재 수집

- 플래너 작성 화면 스크린샷 5장 이상 캡처 (블로그용)
- 사이트 도시 비교 데이터에서 "놀라운 수치" 5개 뽑기 (예: 다낭 평균 월세, 호치민 대비 식비 차이)
- 네이버 카페 "해외이주", "베트남생활", "다낭정보" 최신 질문 20개 리서치 → 자주 나오는 질문 패턴 파악
- 애필리에이트 클릭 동선 최종 점검: 숙소 페이지 → Agoda 링크 클릭 → 추적 정상 작동 확인

#### Day 3 (수) — 네이버 블로그 1편

**제목:** "내가 직접 써본 다낭 한달살기 플래너 공개합니다"

**구성:**
1. 한달살기 준비가 복잡한 이유 (공감 유도)
2. 럭키다낭 플래너 소개 + 스크린샷 5장
3. 체크리스트 주요 항목 미리보기 (23개 중 5개 공개)
4. "전체 플래너 무료로 써보기" CTA → 사이트 링크

**SEO 키워드:** 다낭한달살기준비, 다낭생활비, 다낭장기체류, 베트남한달살기체크리스트

#### Day 4 (목) — 스레드 5연타

스레드는 짧고 구체적인 숫자가 잘 퍼진다. 각 포스팅 마지막에 사이트 링크 포함.

1. "다낭 한달살기 비용 실제 계산해봤습니다 (2026년 기준)" — 사이트 예산 데이터 인용
2. "다낭 vs 나트랑 생활비 차이: 월 OO만원" — Day 2에 수집한 실제 비교 수치로 채울 것 (사이트 /compare 페이지 데이터 기준)
3. "한달살기 전 꼭 해야 할 체크리스트 23가지" — 플래너 링크
4. "숙소 유형별 다낭 월세 비교 (호텔/아파트/게스트하우스)" — 디렉토리 링크
5. "베트남 5개 도시 중 한달살기 비용 가장 저렴한 곳은?" — 도시비교 링크

#### Day 5 (금) — 기존 커뮤니티 참여 + 업체 컨택

**커뮤니티 참여:**
- 네이버 카페 "해외이주·베트남생활" 관련 질문에 성실하게 답변 (최소 5개)
- 답변 마지막에 자연스럽게 사이트 언급: "저는 럭키다낭 사이트에서 비교해봤는데..."
- 카카오 오픈채팅 "다낭한달살기" 그룹 참여 후 사이트 소개

**업체 컨택 (1-2곳):**
- 다낭 현지 한국인 대상 업체 (부동산, 어학원, 마사지 등) 직접 연락
- "지금은 무료로 올려드릴게요. 트래픽 생기면 유료 전환 제안드릴게요" 방식으로 접근
- 목표: 디렉토리에 실제 업체 1-2개 추가

#### Day 6 (토) — 네이버 블로그 2편

**제목:** "다낭 vs 나트랑 한달살기 비교: 실제 데이터로 따져봤습니다"

**구성:**
1. 사이트 도시비교 페이지 스크린샷 활용
2. 숙소비/식비/교통비 항목별 실제 수치 비교
3. 어떤 사람에게 어느 도시가 맞는지 결론
4. 플래너로 직접 계획 세우기 CTA → 사이트 링크

**SEO 키워드:** 나트랑vs다낭, 베트남한달살기비교, 나트랑장기체류

#### Day 7 (일) — 데이터 확인 + 방향 결정

확인할 지표:
- 유입 경로: 블로그 vs 스레드 vs 직접 접속
- 플래너 실제 사용 여부 (체크리스트 체크 이벤트)
- 커뮤니티 자발적 가입자 수
- 애필리에이트 링크 클릭 수
- 디렉토리 페이지 체류 시간

결정 기준:
- 플래너 클릭 > 커뮤니티 가입 → 도구 중심으로 2주차 집중
- 커뮤니티 가입 > 플래너 클릭 → 커뮤니티 콘텐츠 강화로 2주차 집중
- 애필리에이트 클릭 높음 → 디렉토리 + 추천 기능 우선 개발

### 수익 모델 우선순위

| 순위 | 모델 | 조건 | 현황 |
|------|------|------|------|
| 1 | 애필리에이트 (Agoda/Booking) | 트래픽만 있으면 즉시 수익 | affiliate_clicks 테이블 존재, 링크 내장됨 |
| 2 | 업체 광고 (디렉토리 유료 노출) | 디렉토리 트래픽 충분해지면 | 업체 등록 기능 존재, 수익화 미적용 |
| 3 | 상품 직접 판매 | 사용자 신뢰도 쌓인 후 | 주문 시스템 구축됨 |

## Part 2. 플래너 기능 강화

### 목적

현재 플래너는 기능은 갖춰져 있으나 사용자가 계속 돌아올 이유와 공유할 이유가 없다. 5가지 기능을 추가해 재방문율, 바이럴, 애필리에이트 전환을 높인다.

### 기능 A: 플랜 공유 링크

**설명:** 완성한 플랜에 고유 공개 URL을 생성해 SNS/블로그에 공유할 수 있게 한다.

**DB 변경:**
- `planner_plans` 테이블 신규 생성
  - `id` UUID (공개 slug로 사용)
  - `user_id` INTEGER (nullable — 비로그인도 저장 가능)
  - `session_id` TEXT (비로그인 식별용)
  - `title` TEXT (예: "다낭 30일 플랜")
  - `data` JSONB (현재 localStorage의 PlannerData 구조 그대로)
  - `is_public` BOOLEAN DEFAULT false
  - `created_at` TIMESTAMPTZ
  - `updated_at` TIMESTAMPTZ

**API:**
- `POST /api/planner/plans` — 플랜 저장, UUID 반환
- `GET /api/planner/plans/:id` — 공개 플랜 조회
- `PATCH /api/planner/plans/:id` — 플랜 수정 (본인만)

**UI:**
- 플래너 상단에 "공유하기" 버튼
- 클릭 시 `is_public = true`로 플랜 저장 → `https://luckydanang.com/planner/share/:uuid` URL 클립보드 복사 + "링크가 복사되었습니다" 토스트
- 공개 페이지는 읽기 전용으로 렌더링, 하단에 "나도 플랜 만들기" CTA
- 이미 저장된 플랜에서 공유하기 재클릭 시 저장 없이 기존 URL 바로 복사

### 기능 B: 다른 사람 플랜 탐색

**의존성:** 기능 A(`planner_plans` 테이블 및 공유 API) 완료 후 구현 가능.

**설명:** 공개된 플랜들을 조건(도시, 인원, 예산)으로 탐색한다. Pinterest식으로 구경하다가 자신의 플랜을 만들게 유도한다.

**API:**
- `GET /api/planner/plans/public?city=다낭&party=커플&budget_max=3000000` — 필터링된 공개 플랜 목록

**UI:**
- `/planner/explore` 페이지 신규
- 카드 형태: 도시 · 기간 · 인원 · 예산 · 체크리스트 완료율 표시
- 카드 클릭 → 해당 플랜 공개 페이지
- "이 조건으로 나도 플래너 시작하기" 버튼 → 해당 설정값 자동 입력된 플래너로 이동

### 기능 C: 조건 맞춤 추천

**설명:** 플래너의 도시 + 예산 + 인원 정보를 기반으로 디렉토리 숙소와 업체를 추천한다. 애필리에이트 링크와 직접 연결된다.

**로직:**
- 도시 일치 listings 중 `accommodation` 카테고리 필터
- 예산 범위에 맞는 가격대 (listings.price_range 활용)
- 상위 3개 추천, 각 카드에 Agoda/Booking 애필리에이트 링크 포함

**UI:**
- 플래너 내 "추천 숙소" 섹션 (Housing 탭 안에 위치)
- "현재 조건에 맞는 추천 숙소" 카드 3개
- 카드 클릭 → affiliate_clicks 기록 → 외부 예약 링크 이동

**API 변경 없음:** 기존 `/api/listings?category=accommodation&city=다낭` 활용

### 기능 D: D-day 카운트다운 + 이메일 알림

**설명:** 출발일을 기준으로 D-day를 표시하고, 미완료 체크리스트 항목을 이메일로 알린다. 사용자가 플래너를 계속 열게 만든다.

**DB 변경:**
- `planner_plans` 테이블의 `data.startDate` 활용 (기존 필드)
- `planner_reminders` 테이블 신규
  - `plan_id` UUID
  - `user_id` INTEGER
  - `email` TEXT
  - `remind_at` TIMESTAMPTZ (출발 D-30, D-14, D-7)
  - `sent` BOOLEAN DEFAULT false

**알림 타이밍:** D-30, D-14, D-7

**이메일 내용 (Resend 활용):**
- 제목: "다낭 출발 D-14! 아직 [N]개 체크리스트가 남았어요"
- 미완료 항목 목록
- 플래너 바로가기 링크

**UI:**
- 플래너 상단에 D-day 배지 표시 (예: "출발 D-23")
- 최초 저장 시 이메일 입력 모달 (선택 사항)

**서버 작업:**
- `node-cron` 패키지 사용 (Express 서버 상시 실행 구조에 적합)
- `server/src/jobs/planner-reminders.ts` 파일 신규 생성
- 매일 오전 9시(KST) 실행: `planner_reminders` 테이블에서 `remind_at <= now() AND sent = false` 조회 → Resend로 이메일 발송 → `sent = true` 업데이트
- `server/src/index.ts`에서 앱 시작 시 job 등록

### 기능 E: 실시간 예산 시뮬레이터

**설명:** 기존 예산 입력 UX를 슬라이더 방식으로 개선한다. 숙소/식비/교통/기타 비중을 슬라이더로 조절하면 항목별 금액과 총 예산이 실시간 계산된다. 사이트의 도시별 평균 데이터를 기준값으로 제공한다.

**데이터 기준값 (하드코딩 또는 site_settings):**
```
다낭 기준 (30일) — 운영자가 사이트 비교 데이터 기준으로 검토 후 확정 필요:
- 숙소: 월 800,000원 (아파트 기준, 임시값)
- 식비: 월 600,000원 (임시값)
- 교통: 월 150,000원 (임시값)
- 기타: 월 250,000원 (임시값)
```
도시별 기준값은 `site_settings` 테이블에 JSON으로 저장해 관리자 페이지에서 수정 가능하게 구현하는 것을 권장.

**UI:**
- 플래너 예산 탭을 슬라이더 UI로 교체
- 총 예산 입력 → 각 항목 비율 슬라이더 (합계 100%)
- 우측에 실시간 항목별 금액 표시
- "다낭 평균 기준으로 자동 설정" 버튼
- 실제 지출 입력란은 기존 유지

## 구현 범위 정리

| 항목 | 난이도 | 우선순위 |
|------|--------|----------|
| 플랜 공유 링크 (A) | 중 | 높음 — 바이럴 핵심 |
| 실시간 예산 시뮬레이터 (E) | 낮 | 높음 — 프론트엔드만 |
| 조건 맞춤 추천 (C) | 낮 | 높음 — 기존 API 재활용 |
| D-day 카운트다운 UI (D) | 낮 | 중간 — 프론트엔드만 |
| D-day 이메일 알림 (D) | 중 | 중간 — cron 필요 |
| 다른 사람 플랜 탐색 (B) | 중 | 중간 — A 완료 후 |

**권장 구현 순서:** E → C → A → D(UI) → D(이메일) → B
